# NetSentry v2.1  Wazuh Mini-SIEM Integration

## Goal

NetSentry v2.1 adds Wazuh as a mini-SIEM layer on top of the NetSentry gateway.

The goal is to collect and correlate security-relevant logs from the gateway, IDS engines, web stack, firewall, and system services.

This phase does not replace Snort or Suricata. Wazuh is added as the centralized visibility and alerting layer.

---

## Current NetSentry Security Stack

Current always-on components:

```text
NetSentry Gateway
   AP / DHCP / DNS / routing / firewall / NAT
   Nginx + Flask dashboard
   AdGuardHome DNS filtering
   Snort AP-side IDS
   Suricata AP-side IDS
   Tailscale remote administration
   Wazuh all-in-one SIEM layer
```

## Resource Tuning

The NetSentry gateway has limited hardware, so Wazuh was tuned to reduce load.

Current tuning direction:

- Wazuh indexer heap reduced from 1024m to 512m
- Vulnerability detection disabled or reduced
- Syscheck scanning reduced
- Swap used as safety buffer
- Wazuh dashboard can be stopped if the system becomes laggy

Current target:

- Gateway services remain stable first.
- Wazuh is allowed to run only if it does not break gateway usability.

## Safety Notes

- Do not commit Wazuh credentials.
- Do not commit Wazuh generated certificates or private keys.
- Do not commit full Wazuh logs.
- Do not expose Wazuh dashboard broadly.
- Do not expose Wazuh indexer ports.

Wazuh dashboard access should stay limited to:

- Admin laptop
- Tailscale remote admin network

## Success Criteria

This phase is successful when:

- Wazuh dashboard works on port 8443
- NetSentry dashboard still works on port 443
- Wazuh collects Suricata eve.json
- Wazuh collects Snort alert_fast.txt
- Wazuh collects Nginx logs
- Wazuh collects auth/syslog events
- Firewall logs are added safely with rate limits
- System remains usable after Wazuh tuning
